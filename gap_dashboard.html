📖 使用手冊（怎麼讀這張表、stale 了怎麼補資料）

① 這張表在看什麼

監控 catalog 裡每一張 view 的「新鮮度」：最新一筆資料離今天多遠、各層（Raw/Bronze/Silver/Gold）有沒有料、完整度夠不夠。一列 = 一個 dataset。

② 怎麼判讀一列

  • 狀態燈：🔴 STALE（太舊，要補）／⚠️ WARN（開始落後）／✅ OK（新鮮）／❓ EMPTY（完全沒料）／ℹ️ INFO（衍生表，跟著上游）。判定門檻依頻率不同（日／月／季），詳見底部圖例。
  • Tier：P0 最重要（粗體置頂）、P2 次要（淡化）。
  • 完整度／填滿條：clamp(1 − lag_days/90, 0, 1)，條完全填滿 = 最新。
  • 延遲：日交易類已 trading-day-aware（週末／盤前不算）。
  • 說明欄：滑鼠停在標題可看「裡面放了什麼」的中文長描述；ⓘ 表示有額外註解。
  • Raw/Bronze/Silver/Gold 欄：該層檔案大小／catalog 列數，空白 = 該層尚無資料。

③ 看到 STALE / WARN 怎麼補資料

  1. 看該列 「建議操作」欄——非 OK 的列會直接給出該跑的 fetch_cmd 指令，照著貼到終端機跑即可。
  2. 不確定該跑哪個增量爬蟲時，可直接叫我「跑增量爬蟲補 <view 名>」，會走 incremental-crawler 流程並自動重生本表。
  3. 原始檔放 bronze/（不可變）；重抓只能往 silver/ 寫，dedup 用 unique(subset=key, keep='last')。

④ 怎麼加／改某張 view 的註解

點上方 📝 編輯註解 按鈕，在彈窗裡逐列填寫低完整度原因等備註，按「儲存全部」會存回 Flask POST /api/gap_comments（落地 meta/gap_comments.json）。需 Search UI 後端在跑才能存。

⑤ 怎麼重生這張 dashboard

補完資料後，在 repo 根目錄跑：
  • python scripts/gap_report.py --format html — 重生本頁（同時寫 docs/ 與 docs-site/ mirror）。
  • python scripts/gap_report.py --format all — 另含 text + json 輸出。
⚠ DuckDB 鎖：若 duckdb -ui CLI 正開著會鎖住整個 catalog，重生前先確認沒人佔用。

排序：完整度由高到低；同分時 Tier P0 在上。

狀態判定： 日交易 0-1d=OK / 2-5d=WARN / >5d=STALE； 月度 0-15d=OK / 15-45d=WARN / >45d=STALE； 季度 0-60d=OK / 60-120d=WARN / >120d=STALE； 事件型（未來日）以年齡判斷；衍生表（derived）繼承上游 → INFO。
